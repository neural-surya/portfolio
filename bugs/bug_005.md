## Summary

When `return_verification=false`, the `initiate_return` tool processes return requests for orders over $500 without requiring identity verification. The $500 verification threshold is the business's fraud prevention gate — bypassing it allows fraudulent high-value returns without any account validation.

---

## Business Rule

Per Helix Store policy: orders with a total value exceeding **$500** require identity verification before a return can be initiated. The `initiate_return` tool enforces this via the `return_verification` feature flag.

## Impact

- **Financial fraud:** Attacker can initiate returns for high-value orders that don't belong to them, potentially receiving a refund for a $1,299.99 laptop they didn't buy.
- **Operational cost:** Fraudulent returns increase reverse logistics costs and reduce revenue.
- **Systemic risk:** If the flag is inadvertently set to `false` in production, all high-value return fraud protection is silently removed.

---

## Root Cause

In `app/tools/order.py`, the verification check is gated on the `return_verification` flag:

```python
if get_flag("return_verification") and o["total"] > _RETURN_VERIFICATION_THRESHOLD:
    return {
        "status": "verification_required",
        "requires_verification": True,
        "message": "Identity verification required for returns over $500.",
    }
```

When `return_verification=False`, the condition short-circuits and the return is processed regardless of the order total.

---

## Mitigation

Ensure `return_verification=true` in all environments:
```bash
curl -s -X POST http://localhost:8080/api/flags \
  -H "Content-Type: application/json" \
  -d '{"flags": {"return_verification": true}}'
```

For production hardening, the flag should be removed from the API and the verification gate should always be enforced, with no runtime toggle.

## Suggested Fix Direction

```
Fix direction: Remove return_verification from the public flag API entirely. The
verification gate should be a hardcoded constant in initiate_return, not a
runtime toggle. The threshold constant (_RETURN_VERIFICATION_THRESHOLD = 500.0)
should be configurable only via environment variable at deploy time, not via API.
```

## Stakeholder Impact

> HelixBot can be tricked into processing high-value returns without verifying who is making the request. Normally, orders over $500 require the customer to verify their identity first — this prevents people from returning orders that don't belong to them. When this verification feature is disabled (whether accidentally or by configuration error), anyone who knows an order ID can initiate a return for any high-value order. For a $1,299 laptop, this represents direct financial loss to Helix Store.
