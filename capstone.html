Scope Philosophy
What This Capstone Is — and Is Not
💡
The capstone is not a demonstration of everything built across 9 sessions. It is a single, coherent test engagement — strategy to CI/CD — executed against one real target. Think of it as something you would hand to a hiring manager.
⏱
Time reality: Weeknights ~2h/night (10h) + weekend ~10h = ~20 hours total. Every deliverable below fits within that budget. Anything beyond this scope belongs in your session exercises, not here.
✓
Reuse aggressively. ~80% of the code you need already exists in sessions 1–9. The capstone is about integration and communication, not building new things from scratch.

Test Strategy Document

  • The 3 highest-risk areas and the rationale for each
  • Which testing techniques address which risks
  • Acceptance criteria — what passing looks like

2
PromptFoo Test Suite
The core deliverable of the track. 20+ test cases covering 4 of the 7 challenges, run across 2 providers.
  • 2 providers minimum (e.g., gpt-4o-mini + Claude Haiku)
  • At least 4 deterministic assertion types (contains, not-contains, regex, json-schema)
  • At least 5 model-graded assertions (llm-rubric with custom criteria)
  • CSV test data file for parameterised inputs
  • Coverage: hallucination, prompt injection, context limits, bias, non-determinism (5 of 7 challenges)
  • Non-determinism: toggle high_temperature flag — run same query 5× and measure variance
  • Bias: toggle bias_mitigation flag — parameterise queries with demographic descriptors via CSV

3
Red Team Report
PromptFoo red team only — Large config, 100+ adversarial tests, OWASP mapped. Do not attempt PyRIT or Garak in the capstone; environment setup cost is too high for 1 week.
  • Large config: minimum 100 adversarial tests
  • Plugins: promptInjection, jailbreak, systemPromptLeakage, piiLeakage, hallucination, ragPoisoning
  • For ragPoisoning: enable rag_poisoning flag before the run — injects misleading docs into retrieval (LLM09 Supply Chain)
  • Findings table: OWASP ID · severity · reproduction step · remediation
  • Label findings as "security-aware observations", not a penetration test

4
Evaluation — Pick One
Choose the option that fits your target application. Do not do both — 3 hours is enough for one done well.

Option A — RAGAS Evaluation

  • Minimum 10 QA pairs
  • Report: Faithfulness, Answer Relevancy, Context Precision, Context Recall
  • Pass/fail against thresholds (F≥0.85, AR≥0.80, CP≥0.75, CR≥0.75)
  • One-paragraph deployment recommendation
Pick this if: app has a RAG / knowledge base component

Option B — Function Calling Contracts

  • Pydantic schema for each tool the LLM calls
  • Cases: valid call, missing field, wrong type, injection attempt
  • Validate against OpenAPI spec if available
  • Pass/fail summary with failure root cause
Pick this if: app relies on tool use / external APIs

5
GitHub Actions Pipeline
One workflow file. Smoke tests on every push. 3–4 gates that enforce quality thresholds. Start clean — do not merge your 8 session workflows, that will take more than 2 hours.
  • Single workflow file (capstone-pipeline.yml)
  • Smoke tests on push, full suite as manual trigger
  • 3–4 gates: safety pass rate ≥95% + PromptFoo pass rate ≥80% + your evaluation gate
  • Build fails on gate breach — exit code 1 enforcement

6
Bug Report Portfolio + Stakeholder Summary
Minimum 4 high-signal bugs from exploratory testing, plus a half-page plain-English summary written for a product manager or executive — no jargon, no OWASP codes. This answers the most common interview question in AI QA.
  • Title · AI vulnerability category · severity (P0–P4) — categories include knowledge base poisoning
  • Exact prompts used to reproduce
  • Actual vs expected output
  • Reproduction rate (e.g., "4 of 5 attempts")
  • OWASP LLM mapping where applicable
  • Flag state in reproduction steps — use POST /api/flags to make every bug deterministic
  • Stakeholder summary: top 3 findings in plain language + ship / don't ship / ship with conditions recommendation
